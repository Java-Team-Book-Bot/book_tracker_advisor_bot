# Требования к проекту "Book Tracker and Advisor Bot"

## 1. Введение
**Book Tracker and Advisor Bot** — это Telegram-бот для учета прочитанных книг, ведения списков чтения и получения интеллектуальных рекомендаций. Бот интегрируется с Google Books API для поиска метаданных и использует GigaChat API для генерации персонализированных советов.

## 2. Общие требования:
- **Стек**: Java 25, Spring Framework 7.
- **Ограничение**: Использование Spring Boot **строго запрещено**.
- **Сборка**: Gradle, упаковка в Fat JAR.
- **Развертывание**: Docker, Docker Compose.

## 3. Функциональные требования

### 3.1. Общие требования

#### 3.1.1. Авторизация
- **Через Telegram**: Автоматическая идентификация пользователей на основе уникального идентификатора чата (`chat_id`). Дополнительная регистрация не требуется.
- **Через HTTP**: Доступ к защищенным административным REST-эндпоинтам посредством проверки API-ключа (передается в HTTP-заголовке).

#### 3.1.2. Роли
- **User**: Базовый пользователь. Взаимодействует с системой исключительно через интерфейс Telegram-бота.
- **Admin**: Администратор системы. Имеет доступ к защищенным HTTP-эндпоинтам для мониторинга и управления данными.

#### 3.1.3. Системные HTTP-эндпоинты
- **GET /healthcheck** (Публичный эндпоинт):
  - **Авторизация**: Не требуется.
  - **Возвращает**: Текущий статус сервера (например, `{"status": "UP"}`) и массив со списком авторов проекта (студентов).
- **GET /users** (Только для администраторов):
  - **Авторизация**: Требуется валидный API-ключ. При его отсутствии или несовпадении возвращается статус `401 Unauthorized`.
  - **Возвращает**: Полный список всех зарегистрированных пользователей системы.

#### 3.1.4. Обработка невалидного ввода
При получении любых нетекстовых сообщений (фото, стикеры, файлы, голосовые) бот отвечает: *"Пожалуйста, используйте текстовые команды. Введите /help для списка команд"*.

### 3.2. Управление предпочтениями

Пользователь может задать свои литературные вкусы (любимые жанры, авторы, тематики) для персонализации рекомендаций, генерируемых LLM-советником. Данные привязываются к `chat_id` пользователя и сохраняются в базе данных (MongoDB). Максимальный лимит для одного пользователя — 15 записей.

* **Добавление предпочтения**
  * **Команда:** `/preferences add <текст>`
  * **Примеры использования:** `/preferences add Киберпанк`, `/preferences add Братья Стругацкие`, `/preferences add Научная фантастика 20 века`
  * **Ограничение на длину запроса:** максимум - 50 символов (не считая самой команды).
  * **Ожидаемые ответы бота:**
    * *При успехе:* <<Предпочтение <текст> добавлено!
    * *При превышении лимита (более 15):* <<Достигнут лимит предпочтений (15). Пожалуйста, удалите неактуальные записи с помощью `/preferences remove <номер>`.>>
    * *При неверном формате (отсутствие текста):* <<Укажите жанр или автора. Пример использования: `/preferences add антиутопия`.>>
    * *При превышении максимальной длины запроса:* <<Ваш запрос длиннее 50 символов, пожалуйста, сократите его и повторите попытку.>> 

* **Просмотр списка предпочтений**
  * **Команда:** `/preferences show`
  * **Ожидаемые ответы бота:**
    * *При наличии записей:* Выводится нумерованный список всех сохраненных предпочтений.
    * *Если список пуст:* <<У вас пока нет сохраненных предпочтений. Добавьте их с помощью команды `/preferences add <жанр/автор>`.>>

* **Удаление предпочтения**
  * **Команда:** `/preferences remove <номер>` (где `<номер>` — это порядковый номер записи из списка `/preferences show`).
  * **Пример использования:** `/preferences remove 2`
  * **Ожидаемые ответы бота:**
    * *При успехе:* <<Предпочтение 'Братья Стругацкие' успешно удалено.>>
    * *При указании несуществующего номера:* <<Предпочтение с таким номером не найдено. Пожалуйста, сверьтесь с вашим списком: `/preferences show`.>>
    * *При неверном формате ввода (например, передана буква вместо цифры):* <<Укажите корректный номер из списка. Пример: `/preferences remove 1`.>>

### 3.3. Список чтения (Tracker)

Основной функционал для учета пользовательской библиотеки. Данные о сохраненных книгах, их текущих статусах и прогрессе привязываются к `chat_id` пользователя и хранятся в базе данных (MongoDB). Максимальный лимит списка для одного пользователя — 200 книг.

* **Добавление книги в список**
  * Добавление происходит через интерактивные кнопки `Добавить в список` под результатами поиска (команды `/search` или `/random`), где пользователю предлагается выбрать начальный статус: `Хочу прочитать` (to_read), `Читаю` (reading) или `Прочитано` (finished).
  * **Ожидаемые ответы бота:**
    * *При достижении лимита количества книг в списке:* <<Достигнут лимит списка (200 книг). Пожалуйста, удалите неактуальные записи с помощью команды `/remove <номер>`.>>
    * *При наличии выбранной книги в списке:* <<Эта книга уже есть в списке со статусом <статус книги>.>>

* **Просмотр списка чтения (с фильтрацией)**
  * **Команда:** `/list [статус]` (параметр статуса опционален: `to_read`, `reading`, `finished`).
  * **Примеры использования:** `/list` (покажет весь список), `/list reading` (покажет только книги, которые читаются сейчас).
  * **Ожидаемые ответы бота:**
    * *При наличии записей:* Выводит нумерованный список книг с указанием их внутреннего `<номер>`, автора, названия и текущего статуса. Дополнительно выводятся контекстные данные в зависимости от статуса:
      * Для статуса `reading`: отображается сохраненный прогресс (например, текущая страница или глава).
      * Для статуса `finished`: отображается выставленная пользователем оценка (например, `9/10`) и текст личной заметки, если они были ранее добавлены.
    * *Если список пуст:* <<Ваш список чтения пока пуст. Найдите интересные книги с помощью команды `/search <запрос>`.>>
    * *Если по фильтру ничего не найдено:* <<У вас нет книг со статусом 'reading'.>>
    * *Если указан некорректный статус:* <<Укажите корректный статус. Доступные варианты: to_read, reading, finished.>>

* **Изменение статуса книги**
  * **Команда:** `/status <номер> <новый_статус>`
  * **Примеры использования:** `/status 15 reading`, `/status 12 finished`
  * **Ожидаемые ответы бота:**
    * *При успехе:* <<Статус книги 'Дюна' успешно изменен на 'Читаю'.>>
    * *При указании несуществующего ID:* <<Книга с ID 15 не найдена в вашем списке. Проверьте ID через команду `/list`.>>
    * *При неверном формате статуса:* <<Укажите корректный статус. Доступные варианты: to_read, reading, finished.>>

* **Обновление прогресса чтения**
  * **Команда:** `/progress <номер> <значение>` (значением может быть номер страницы или название главы).
  * **Примеры использования:** `/progress 15 120`, `/progress 15 Глава 5`
  * **Ограничение длины значения прогресса:** максимум 50 символов, не включая текст команды.
  * **Ожидаемые ответы бота:**
    * *При успехе:* <<Прогресс для книги 'Дюна' обновлен: 120.>>
    * *Ошибка статуса:* <<Обновлять прогресс можно только для книг в статусе 'Читаю'. Текущий статус этой книги: 'Хочу прочитать'.>>
    * *При указании несуществующего ID:* <<Книга с таким ID не найдена в вашем списке.>>
    * *При превышении ограничения на длину запроса:* <<Ваш запрос длиннее 50 символов. Пожалуйста, сократите его и повторите попытку.>>

* **Удаление книги из списка**
  * **Команда:** `/remove <номер>` (где `<номер>` — порядковый номер книги в выводе команды `/list`).
  * **Пример использования:** `/remove 3`
  * **Ожидаемые ответы бота:**
    * *При успехе:* «Книга 'Дюна' успешно удалена из вашего списка.»
    * *При указании несуществующего номера:* «Книга с таким номером не найдена. Проверьте список через команду `/list`.»

### 3.4. Поиск книг

Поиск осуществляется через интеграцию с внешним сервисом Google Books API с использованием `RestClient`.

* **Поиск по явно заданному запросу**
  * **Команда:** `/search <жанр, автор, название или ключевые слова>`
  * **Примеры использования:** `/search Фрэнк Герберт`, `/search киберпанк антиутопия`
  * **Ограничение на длину запроса:** максимум 100 символов, не учитывая текст команды.
  * **Ожидаемые ответы бота:**
    * *При успехе (Режим списка):* Бот отправляет единое сообщение со списком до 5 найденных книг (содержит автора, название и год издания). Под сообщением выводятся inline-кнопки с порядковыми номерами книг `[1]` – `[5]`, а также кнопка `[Далее]` для загрузки следующей партии результатов и `[Назад]`, чтобы вернуться к предыдущему списку.
    * *Переход к карточке (Детальный вид):* При нажатии на кнопку с номером, текущее сообщение изменяется на подробную информационную карточку выбранной книги, с полным названием, авторами и описанием с Google Books. 
    * *Действия с карточкой:* Под детальной карточкой пользователю доступны интерактивные кнопки: <<Добавить в список>> (с дальнейшим выбором статуса: *Хочу прочитать / Читаю / Прочитано*), <<Саммари от LLM>> и кнопка <<Назад к списку>> (которая возвращает сообщение к предыдущему виду списка из 5 книг).

* **Поиск на основе предпочтений (без параметров)**
  * **Команда:** `/search` (без текста запроса)
  * **Логика работы:** Согласно требованиям, если критерии не заданы, бот обращается к базе данных (MongoDB), извлекает сохраненные предпочтения пользователя и формирует из них запрос к Google Books API.
  * **Ожидаемые ответы бота:**
    * *При наличии предпочтений:* Ответы бота аналогичны ответам при поиске с запросом.
    * *Если список предпочтений пуст:* <<Критерии поиска не заданы, а ваш список предпочтений пуст. Уточните запрос (например, `/search Дюна`) или добавьте любимые жанры через `/preferences add <текст>`.>>

* **Запрос случайной книги**
  * **Команда:** `/random`
  * **Логика работы:** Бот подбирает одну абсолютно случайную книгу из базы Google Books.
  * **Ожидаемые ответы бота:**
    * Бот сразу отправляет сообщение в виде подробной информационной карточки найденной книги (содержит Название, Автора, Год издания и Описание).
    * Под карточкой выводятся интерактивные кнопки: <<Добавить в список>> (с дальнейшим выбором статуса: *Хочу прочитать / Читаю / Прочитано*) и <<Саммари от LLM>>.

### 3.5. LLM-советник (GigaChat)

Интеллектуальный модуль на базе внешнего GigaChat API, выступающий в роли персонального литературного помощника. Для всех запросов к LLM реализована отправка промежуточного статуса (например, *<<Генерирую ответ...>>*), так как генерация может занимать до 30 секунд.

* **Персональные рекомендации**
  * **Команда:** `/recommend`
  * **Логика работы:** Бот извлекает из БД (MongoDB) историю книг пользователя (статусы `finished` и `reading`) и сохраненные предпочтения. Эти данные упаковываются в системный промпт и отправляются в GigaChat для генерации подборки из 3-х книг.
  * **Ожидаемые ответы бота:**
    * *При успехе:* Отформатированный текст от LLM с рекомендациями и кратким обоснованием (почему именно эта книга подходит пользователю).
    * *При недостатке данных:* <<Для точных рекомендаций мне нужно лучше узнать ваши вкусы. Оцените пару книг или заполните предпочтения через команду `/preferences add`.>>

* **Поиск похожих книг**
  * **Команда:** `/similar <номер>` (где ID — идентификатор книги из списка пользователя).
  * **Логика работы:** Бот отправляет запрос к LLM с просьбой подобрать произведения, похожие по стилю, атмосфере и жанру на указанную книгу.
  * **Ожидаемые ответы бота:**
    * *При успехе:* Текстовый список из 3-х произведений с объяснением их сходства с оригиналом.
    * *При неверном ID:* <<Книга с таким ID не найдена в вашем списке.>>

* **Генерация саммари (краткого обзора)**
  * **Интерфейс:** Вызывается нажатием интерактивной кнопки `[🤖 Саммари от LLM]` под детальной карточкой книги (при поиске или просмотре списка).
  * **Логика работы:** Бот передает название и автора книги в GigaChat с задачей составить выжимку сюжета и ключевых идей произведения.
  * **Ожидаемые ответы бота:** Текстовое сообщение с кратким обзором книги. Если генерация для данной книги уже запущена пользователем, повторные нажатия игнорируются с ответом: *<<Саммари уже генерируется, пожалуйста, подождите.>>*

* **Свободный вопрос советнику (Free-form question)**
  * **Команда:** `/ask <текст вопроса>` (длина текста строго ограничена 500 символами).
  * **Примеры использования:** `/ask Что почитать в стиле киберпанка, но чтобы действие происходило в космосе?`
  * **Логика работы:** Пользователь задает произвольный вопрос. Системный промпт ограничивает LLM ролью литературного критика, чтобы бот не отвечал на вопросы, не связанные с книгами.
  * **Ожидаемые ответы бота:**
    * *При успехе:* Развернутый ответ LLM на вопрос пользователя.
    * *При превышении лимита:* <<Ваш вопрос слишком длинный (максимум 500 символов). Пожалуйста, сформулируйте его короче.>>

### 3.5.1. Обработка длительных операций (UX)
Так как генерация ответов от LLM может занимать до 30 секунд, для обеспечения качественного пользовательского опыта применяется механизм промежуточных статусов:
1. **Промежуточное сообщение:** Сразу после получения команды (например, `/recommend` или нажатия кнопки "Саммари") бот отправляет сообщение: *«⏳ Генерирую ответ, пожалуйста, подождите...»*.
2. **Блокировка дубликатов:** Пока висит это сообщение, новые аналогичные запросы от этого же пользователя блокируются. При попытке вызвать команду повторно бот отправляет предупреждение (которое удаляется через 3 секунды): *«Ваш предыдущий запрос еще обрабатывается»*.
3. **Редактирование при успехе:** Как только получен ответ от GigaChat API, бот изменяет промежуточное сообщение на итоговый сгенерированный текст.
4. **Редактирование при таймауте/ошибке:** Если истекает таймаут в 30 секунд или GigaChat возвращает ошибку, промежуточное сообщение изменяется на: *«Не удалось сгенерировать ответ. Пожалуйста, попробуйте позже.»*

### 3.6. Бонусные возможности

Реализация дополнительных функций для превращения бота в полноценный цифровой читательский дневник и интеграции с продвинутыми архитектурными паттернами (очереди сообщений).

* **Рейтинги и личные заметки**
  * **Команды:** `/rate <номер> <1-10>` и `/note <номер> <текст заметки>`
  * **Логика работы:** Пользователь может оценить книгу и оставить к ней короткий комментарий. Оценка и текст сохраняются в документе книги в MongoDB и в дальнейшем используются в качестве контекста (промпта) для LLM-советника, чтобы сделать рекомендации максимально точными.
  * **Ограничения:**
    * Оценивать и комментировать можно только книги, находящиеся в статусе <<Прочитано>> (`finished`). При попытке оценить книгу из другого списка бот выдаст ошибку.
    * Длина личной заметки ограничена 300 символами.
  * **Отображение:** При вызове команды `/list finished` к карточке книги добавляются строки с оценкой (например, `9/10`) и текстом заметки.

* **Любимые цитаты**
  * **Команда добавления:** `/quote add <ID_книги> <текст цитаты>`
  * **Команды просмотра:** 
    * `/quotes <ID_книги>` - выводятся все цитаты для выбранной книги.
    * `/quotes` - выводятся все цитаты для всех книг в формате: ID_книги, Название книги, автор: <список цитат>.
  * **Логика работы:** Цитаты сохраняются в виде массива строк внутри объекта книги в БД. Длина одной цитаты ограничена 500 символами. Максимальное количество цитат для одной книги: 10.

* **Шеринг списка чтения**
  * **Команда:** `/share`
  * **Логика работы:** Бот генерирует уникальную ссылку вида `t.me/BookAdvisorBot?start=list_<chat_id>` и отправляет её пользователю для пересылки друзьям.
  * **Отображение (Read-only режим):** При переходе по ссылке бот распознает параметр `list_...` и выводит чужой список книг **в точно таком же интерфейсе, как при поиске (см. п. 3.4)** — в виде единого сообщения с пагинацией `[Далее]` и inline-кнопками для перехода к детальной карточке книги.
  * **Взаимодействие:** В режиме просмотра чужого списка редактирование статусов или удаление заблокировано. Однако в детальной карточке книги пользователю доступна кнопка <<Добавить в свой список>>, что позволяет легко забирать книги друзей в свою личную библиотеку.

* **Напоминания о чтении (ZeroMQ Integration)**
  * **Команды:** `/remind on <ЧЧ:ММ>` (включить ежедневное напоминание на заданное время) и `/remind off` (выключить).
  * **Архитектура реализации:** 1. Системный планировщик Spring (`@Scheduled`) ежеминутно проверяет БД на наличие пользователей, которым нужно отправить уведомление в текущую минуту.
    2. Планировщик формирует задачи на отправку и асинхронно помещает их в брокер сообщений **ZeroMQ**.
    3. Отдельный сервис-слушатель (worker) вычитывает очередь ZeroMQ и отправляет уведомления через Telegram API (`RestClient`).

### 3.7. Сценарии использования (Use Cases)
- **Сценарий 1: Поиск и добавление книги.**
    1. Пользователь вводит `/search Dune`.
    2. Бот отправляет HTTP-запрос через `RestClient` к Google Books API.
    3. Бот парсит полученный JSON и формирует сообщение-список (до 5 книг) с кнопками навигации.
    4. Пользователь выбирает книгу, переходит в её карточку и нажимает «Добавить в список».
    5. Бот запрашивает статус (Хочу прочитать / Читаю / Прочитано).
    6. Пользователь выбирает статус, и запись о книге сохраняется в **MongoDB**.
- **Сценарий 2: Запрос рекомендаций.**
    1. Пользователь вводит `/recommend`.
    2. Бот собирает список предпочтений и историю чтения из **MongoDB**.
    3. Бот формирует и отправляет промпт в GigaChat API.
    4. Бот возвращает отформатированный ответ пользователю.

## 4. Нефункциональные требования

### 4.1. Производительность и БД
- Хранение всех данных осуществляется в документо-ориентированной БД **MongoDB**.
- Время отклика бота (без учета LLM) — не более 1 секунды.

### 4.2. Надежность и Retry-логика
- **Google Books API**: При таймауте (>5 сек) выполняется 2 повторные попытки с экспоненциальной задержкой.
- **GigaChat API**: Таймаут ожидания — 30 сек. Повторные попытки **не выполняются**. При ошибке выводится: *"Не удалось сгенерировать ответ. Попробуйте позже"*.

### 4.3. Блокировка дублирующихся запросов
Пока длительный запрос (поиск по Google Books или генерация ответа от GigaChat) находится в процессе выполнения, все новые аналогичные запросы от этого же `chat_id` блокируются. Бот отправляет временное сообщение: *«Ваш предыдущий запрос еще обрабатывается, пожалуйста, подождите»* (сообщение автоматически удаляется через 3-5 секунд, чтобы не засорять историю чата).


### 4.4. Безопасность и Валидация
- **Хранение секретов**: Все ключи (Telegram Token, GigaChat Auth) передаются исключительно через переменные окружения (`.env`). Хранение ключей в коде строго запрещено.
- **Валидация ввода**:
  * Параметры `<номер>` в командах (например, `/remove 2`, `/status 5 reading`) валидируются регулярным выражением `^[1-9]\d{0,2}$` (допускаются только положительные числа до трех знаков, так как лимит списка 200 книг).
  * Параметр `<значение>` для `/progress` допускает только буквы, цифры и пробелы: `^[a-zA-Zа-яА-ЯёЁ0-9\s]{1,50}$`. При наличии спецсимволов запрос отклоняется.
  * Текстовые запросы для LLM (`/ask`, `/search`) экранируются перед вставкой в JSON-тело запроса к GigaChat/Google API для предотвращения инъекций.

## 7. Глоссарий

### 7.1. Технические термины
- **Fat JAR**: Исполняемый архив, включающий скомпилированный код приложения и все его зависимости, позволяющий запускать приложение одной командой `java -jar`.
- **ZeroMQ**: Высокопроизводительный асинхронный брокер сообщений, используемый для управления очередью фоновых задач (напоминаний).
- **Spring Rest Docs**: Инструмент для генерации документации HTTP API на основе успешно пройденных интеграционных тестов.
- **LLM (Large Language Model)**: Большая языковая модель (в проекте — GigaChat), используемая для анализа текста и генерации рекомендаций.

### 7.2. Термины предметной области
- **Tracker (Список чтения)**: Персональная коллекция книг пользователя с отслеживанием статуса (Хочу прочитать / Читаю / Прочитано).
- **Summary (Саммари)**: Краткая выжимка сюжета и ключевых идей книги, сгенерированная искусственным интеллектом.
- **Preferences (Предпочтения)**: Набор любимых жанров, авторов или тематик пользователя, используемый как контекст для повышения точности рекомендаций LLM.
